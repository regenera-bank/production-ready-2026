## 1. VISÃO GERAL E CONTEXTO

### 1.1 O Problema: Complexidade Bancária no Século XXI

Sistemas bancários modernos enfrentam desafios exponenciais:

```
DESAFIOS TÉCNICOS:
├─ Escala: 100K → 10M usuários (crescimento 100x)
├─ Disponibilidade: 99.99% uptime (52 min downtime/ano)
├─ Segurança: PCI-DSS Level 1 + LGPD + BACEN
├─ Performance: <100ms latência p99 em transações
├─ Compliance: 47 regulações simultâneas
└─ Custos: TCO deve cair 30% ao escalar 10x

DESAFIOS DE NEGÓCIO:
├─ Time-to-Market: Features novas em <2 semanas
├─ Inovação: IA/ML sem comprometer estabilidade
├─ Multi-canal: Web + Mobile + API + Open Banking
└─ Diferenciação: UX superior sem sacrificar segurança
```

### 1.2 A Solução: Arquitetura Regenerativa

O Regenera Bank implementa um **Sistema Adaptativo Distribuído** com 5 princípios fundamentais:

#### Princípio 1: Domain-Driven Design (DDD)
```
❌ Arquitetura Monolítica:
   └─ 1 banco de dados, 1 aplicação, 1 ponto de falha

✅ Arquitetura Regenera:
   ├─ 12 domínios de negócio independentes
   ├─ 4 bancos de dados especializados
   └─ Falhas isoladas (blast radius <8% do sistema)
```

#### Princípio 2: Comunicação Híbrida Otimizada
```typescript
enum CommunicationPattern {
  GRPC = "Síncrono, Intra-Serviço, Baixa Latência",
  REST = "Síncrono, Cliente-Servidor, Padrão Web",
  MessageQueue = "Assíncrono, Pub/Sub, Alta Resiliência"
}

// Decisão arquitetural documentada:
const escolherProtocolo = (requisito: string): CommunicationPattern => {
  if (requisito === "Resposta imediata + Alta performance") return GRPC;
  if (requisito === "Exposição pública + Compatibilidade") return REST;
  if (requisito === "Desacoplamento temporal + Escalabilidade") return MessageQueue;
};
```

#### Princípio 3: Segurança em Profundidade (Defense in Depth)
```
Camada 7: Application (WAF + Rate Limiting)
Camada 6: Presentation (XSS + CSRF Protection)
Camada 5: Session (JWT + Refresh Tokens)
Camada 4: Transport (TLS 1.3 + mTLS entre serviços)
Camada 3: Network (Security Groups + Private Subnets)
Camada 2: Data Link (Encryption at Rest - KMS)
Camada 1: Physical (AWS Compliance - SOC 2)
```

#### Princípio 4: Persistência Poliglota
```sql
-- Cada problema tem sua solução otimizada:

PostgreSQL → Transações ACID, integridade referencial
USE CASE: Contas, Saldos, Cartões (consistência crítica)

MongoDB → Flexibilidade de schema, documentos aninhados  
USE CASE: Eventos de Analytics, Logs de AI (schema evolui)

Redis → Cache em memória, <1ms latência
USE CASE: Sessões de usuário, Rate limiting (acesso frequente)

Elasticsearch → Busca full-text, agregações complexas
USE CASE: Auditoria imutável, Compliance logs (queries complexas)
```

#### Princípio 5: Automação Implacável
```yaml
# Nenhuma ação manual em produção. ZERO.
deployment_manual: FORBIDDEN
rollback_manual: FORBIDDEN
configuration_manual: FORBIDDEN

# Tudo é código, tudo é versionado, tudo é auditável:
infrastructure: Terraform (IaC)
pipelines: GitHub Actions (CI/CD)
configurations: ConfigMaps + Secrets (GitOps)
monitoring: Datadog + Prometheus (Observability as Code)
```

## 3. STACK TECNOLÓGICO DETALHADO

### 3.1 Matriz de Decisões Técnicas

| Camada | Tecnologia Escolhida | Alternativas Avaliadas | Razão da Escolha |
|--------|---------------------|------------------------|------------------|
| **Runtime** | Node.js 18 LTS | Java (Spring Boot), Go, Python | • Event-loop eficiente para I/O<br>• Ecossistema NPM maduro<br>• TypeScript = type safety<br>• Time já familiarizado |
| **Framework Backend** | NestJS | Express, Fastify, Koa | • Arquitetura opinionated (DDD)<br>• Decorators = código limpo<br>• DI nativo<br>• gRPC first-class |
| **Framework Frontend** | Next.js 14 | Remix, SvelteKit, Nuxt | • SSR + SSG híbrido<br>• React Server Components<br>• Image optimization<br>• Vercel deploy |
| **Linguagem** | TypeScript 5.x | JavaScript, Flow | • Type safety = menos bugs<br>• Intellisense superior<br>• Refactoring seguro<br>• Strictest mode |

SCENARIO A: Insufficient Balance
├─ Detected at: Step 1.4 (before event publish)
├─ Response: HTTP 400
│  {
│    error: "INSUFFICIENT_BALANCE",
│    required: 15000,
│    available: 8000
│  }
└─ No event published, no state change

SCENARIO B: Account Service Down
├─ Detected at: Step 4 (event not consumed)
├─ RabbitMQ behavior:
│  └─ Message remains in queue
│  └─ Retry after 30s (exponential backoff)
│  └─ Max retries: 5
│  └─ After 5 failures → Dead Letter Queue (DLQ)
├─ PIX Service:
│  └─ Transaction status remains "PENDING"
│  └─ Cron job (every 5 min) checks for stuck transactions
│  └─ Alert sent to ops team
└─ User notification: "Transfer delayed, will retry"

SCENARIO C: Database Transaction Rollback
├─ Detected at: Step 4.3 (constraint violation, deadlock, etc)
├─ PostgreSQL: ROLLBACK automatic
├─ Account Service:
│  └─ Catches exception
│  └─ Publishes "pix.transfer.failed" event
│  └─ Updates pix_transactions.status = 'FAILED'
└─ User receives: "Transfer failed, amount not debited"

SCENARIO D: Partial Failure (sender debited, recipient not credited)
├─ IMPOSSIBLE due to database transaction atomicity
├─ If PostgreSQL crashes mid-transaction:
│  └─ On restart, ROLLBACK of uncommitted transaction
│  └─ Result: Both accounts unchanged
└─ This is why ACID databases are non-negotiable for money

#### 6.2.2 Retry with Exponential Backoff
```typescript
// packages/resilience/src/retry.ts

export class RetryPolicy {
  async execute<T>(
    fn: () => Promise<T>,
    options: {
      maxAttempts: number;
      initialDelay: number;
      maxDelay: number;
      backoffMultiplier: number;
      retryableErrors?: (error: Error) => boolean;
    }
  ): Promise<T> {
    let lastError: Error;
    
    for (let attempt = 1; attempt <= options.maxAttempts; attempt++) {
      try {
        return await fn();
      } catch (error) {
        lastError = error;
        
        // Check if error is retryable
        if (options.retryableErrors && !options.retryableErrors(error)) {
          throw error;
        }
        
        // Don't retry on last attempt
        if (attempt === options.maxAttempts) {
          throw error;
        }
        
        // Calculate delay with exponential backoff
        const delay = Math.min(
          options.initialDelay * Math.pow(options.backoffMultiplier, attempt - 1),
          options.maxDelay
        );
        
        logger.warn(`Attempt ${attempt} failed, retrying in ${delay}ms`, {
          error: error.message,
          attempt,
          maxAttempts: options.maxAttempts
        });
        
        await this.sleep(delay);
      }
    }
    
    throw lastError;
  }

  private sleep(ms: number): Promise<void> {
    return new Promise(resolve => setTimeout(resolve, ms));
  }
}

// Usage:
const retryPolicy = new RetryPolicy();

const result = await retryPolicy.execute(
  () => grpcClient.findUserByEmail({ email: 'test@example.com' }),
  {
    maxAttempts: 3,
    initialDelay: 100,
    maxDelay: 2000,
    backoffMultiplier: 2,
    retryableErrors: (error) => {
      // Only retry on network errors, not business logic errors
      return error.code === 'UNAVAILABLE' || error.code === 'DEADLINE_EXCEEDED';
    }
  }
);

// Retry timeline:
// Attempt 1: Fails → Wait 100ms
// Attempt 2: Fails → Wait 200ms  (100 * 2^1)
// Attempt 3: Fails → Wait 400ms  (100 * 2^2)
// Attempt 4: Success or throw
```

#### 6.2.3 Bulkhead Pattern (Resource Isolation)
```typescript
// packages/resilience/src/bulkhead.ts

export class BulkheadService {
  private readonly pools: Map<string, Pool> = new Map();

  createPool(name: string, options: {
    maxConcurrent: number;
    maxQueue: number;
    timeout: number;
  }) {
    const pool = {
      name,
      maxConcurrent: options.maxConcurrent,
      maxQueue: options.maxQueue,
      timeout: options.timeout,
      running: 0,
      queue: [] as Array<() => void>,
    };

    this.pools.set(name, pool);
  }

  async execute<T>(poolName: string, fn: () => Promise<T>): Promise<T> {
    const pool = this.pools.get(poolName);
    if (!pool) throw new Error(`Pool ${poolName} not found`);

    // Check if we can run immediately
    if (pool.running < pool.maxConcurrent) {
      return this.run(pool, fn);
    }

    // Check if queue is full
    if (pool.queue.length >= pool.maxQueue) {
      throw new Error(`Pool ${poolName} queue is full`);
    }

    // Queue the request
    return new Promise((resolve, reject) => {
      const timeoutId = setTimeout(() => {
        const index = pool.queue.indexOf(queuedFn);
        if (index > -1) pool.queue.splice(index, 1);
        reject(new Error(`Pool ${poolName} timeout`));
      }, pool.timeout);

      const queuedFn = async () => {
        clearTimeout(timeoutId);
        try {
          const result = await this.run(pool, fn);
          resolve(result);
        } catch (error) {
          reject(error);
        }
      };

      pool.queue.push(queuedFn);
    });
  }

  private async run<T>(pool: Pool, fn: () => Promise<T>): Promise<T> {
    pool.running++;
    
    try {
      return await fn();
    } finally {
      pool.running--;
      
      // Process next in queue
      const next = pool.queue.shift();
      if (next) next();
    }
  }
}

// Setup bulkheads:
const bulkhead = new BulkheadService();

bulkhead.createPool('database', {
  maxConcurrent: 20,  // Max 20 concurrent DB connections
  maxQueue: 100,      // Max 100 waiting requests
  timeout: 5000       // 5s timeout for queued requests
});

bulkhead.createPool('external-api', {
  maxConcurrent: 10,
  maxQueue: 50,
  timeout: 3000
});

// Usage:
await bulkhead.execute('database', async () => {
  return userRepository.findByEmail('test@example.com');
});
```

  }

  /**
   * LGPD Art. 18, II - Acesso aos dados
   */
  async exportUserData(userId: string): Promise<UserDataExport> {
    const user = await this.userRepo.findById(userId);
    const accounts = await this.accountRepo.findByUserId(userId);
    const transactions = await this.transactionRepo.findByUserId(userId);

    return {
      personalInfo: {
        name: user.fullName,
        email: user.email,
        cpf: user.cpf,
        phone: user.phone,
        createdAt: user.createdAt
      },
      accounts: accounts.map(a => ({
        accountNumber: a.number,
        balance: a.balance,
        type: a.type
      })),
      transactions: transactions.map(t => ({
        id: t.id,
        type: t.type,
        amount: t.amount,
        date: t.createdAt,
        description: t.description
      }))
    };
  }

  /**
   * LGPD Art. 18, VI - Eliminação dos dados
   */
  async deleteUserData(userId: string): Promise<void> {
    // Soft delete (mantém por 5 anos para compliance fiscal)
    await this.userRepo.softDelete(userId);
    
    // Anonimiza dados não essenciais
    await this.userRepo.anonymize(userId, {
      email: `deleted-${userId}@anonymized.local`,
      phone: null,
      fullName: 'USUARIO EXCLUIDO'
    });

    logger.info('User data deleted (LGPD)', { userId });
  }

  /**
   * LGPD Art. 18, IV - Correção de dados
   */
  async updateUserData(userId: string, updates: Partial<User>): Promise<User> {
    return this.userRepo.update(userId, updates);
  }
}

### 7.3 Audit Trail (Blockchain Service)
```typescript
// apps/services/blockchain-service/src/audit-log.service.ts

import { Client } from '@elastic/elasticsearch';
import * as crypto from 'crypto';

export class AuditLogService {
  constructor(private readonly esClient: Client) {}

  async logTransaction(transaction: AuditableTransaction): Promise<void> {
    const previousBlock = await this.getLatestBlock();
    
    const block: AuditBlock = {
      timestamp: new Date().toISOString(),
      transactionId: transaction.id,
      type: transaction.type,
      userId: transaction.userId,
      data: transaction.data,
      previousHash: previousBlock?.hash || '0',
      hash: '' // Will be calculated
    };

    // Calculate hash (SHA-256)
    block.hash = this.calculateHash(block);

    // Store immutable log in Elasticsearch
    await this.esClient .index({
index: 'audit-logs',
id: transaction.id,
body: block,
refresh: true
});

logger.info('Audit log created', {
  transactionId: transaction.id,
  hash: block.hash
});
}

private calculateHash(block: Omit<AuditBlock, 'hash'>): string {
const data = JSON.stringify(block);
return crypto.createHash('sha256').update(data).digest('hex');
}

async verifyIntegrity(transactionId: string): Promise<boolean> { const block = await this.getBlock(transactionId); if (!block) return false;

// Recalculate hash
const calculatedHash = this.calculateHash({
  ...block,
  hash: undefined
});

// Verify hash matches
if (calculatedHash !== block.hash) {
  logger.error('Audit log integrity violation', { transactionId });
  return false;
}

// Verify chain (previous block)
if (block.previousHash !== '0') {
  const previousBlock = await this.getPreviousBlock(block);
  if (!previousBlock || previousBlock.hash !== block.previousHash) {
    logger.error('Audit log chain broken', { transactionId });
    return false;
  }
}

return true;
}
}

````markdown
# RUNBOOK: High Latency in PIX Service

## Severity: P1 (Critical)
## SLA: 15 minutes to acknowledge, 1 hour to resolve

## Symptoms
- API Gateway latency > 1s (normally 100ms)
- PIX transfer success rate < 95% (normally 99.9%)
- Datadog alert: "pix-service-high-latency"

## Investigation Steps

### 1. Check Service Health (2 min)
```bash
# Check pod status
kubectl get pods -n regenera-production -l app=pix-service

# Check recent logs
kubectl logs -n regenera-production -l app=pix-service --tail=100

# Check metrics
curl https://datadog.regenera.bank/dashboard/pix-service
```

### 2. Identify Bottleneck (5 min)
Common causes:
- [ ] Database connection pool exhausted
- [ ] External API (account-service) slow
- [ ] RabbitMQ queue backlog
- [ ] CPU/Memory throttling
```bash
# Check database connections
kubectl exec -it pix-service-xxx -n regenera-production -- \
  psql -U admin -d regenera -c "SELECT count(*) FROM pg_stat_activity;"

# Check RabbitMQ queue depth
curl -u admin:$RABBITMQ_PASS https://rabbitmq.regenera.bank/api/queues

# Check resource usage
kubectl top pods -n regenera-production -l app=pix-service
```

### 3. Immediate Mitigation (3 min)

**Option A: Scale Up**
```bash
kubectl scale deployment pix-service --replicas=10 -n regenera-production
```

**Option B: Circuit Breaker (if account-service is down)**
```bash
# Enable maintenance mode
kubectl set env deployment/pix-service MAINTENANCE_MODE=true
```

**Option C: Rollback (if recent deploy)**
```bash
kubectl rollout undo deployment/pix-service -n regenera-production
```

### 4. Root Cause Analysis (30 min)
- Review Datadog APM traces for slowest transactions
- Check database explain plans for slow queries
- Review recent code changes in GitHub
- Analyze error logs for patterns

### 5. Permanent Fix
- Deploy hotfix (if code issue)
- Tune database (if query issue)
- Increase resources (if capacity issue)

### 6. Post-Incident
- Update this runbook with learnings
- Create Jira ticket for permanent fix
- Schedule post-mortem meeting (within 48h)
